III. ESTÁNDARES DE CALIDAD — OBLIGATORIOS EN TODO COMPONENTE
==============================================================================

  3.1 SEGURIDAD
  ─────────────

  ▸ Transport:
    - HTTPS obligatorio (HSTS preload donde sea posible)
    - Redirect HTTP→HTTPS en servidor

  ▸ Headers (aplicar en CADA app):
    - X-Frame-Options: DENY
    - X-Content-Type-Options: nosniff
    - Referrer-Policy: strict-origin-when-cross-origin
    - Permissions-Policy: camera=(), microphone=(), geolocation=()
    - Strict-Transport-Security: max-age=63072000; includeSubDomains
    - Content-Security-Policy: específico por app

  ▸ Auth:
    - Sanctum Bearer token para API
    - Token en expo-secure-store (P15)
    - Token en localStorage (P14/P16) — MIGRAR a HttpOnly cookie (backlog)
    - NUNCA exponer secretos en NEXT_PUBLIC_
    - Rate limiting en login (throttle:5,1)

  ▸ CORS:
    - Origins explícitos en .env (no wildcards)
    - Permitir solo: admin., portal., localhost:3000/3001

  ▸ npm Supply Chain:
    - Siempre commitear package-lock.json
    - CI/CD usa npm ci (no npm install)
    - npm audit antes de cada deploy
    - Code cooldown: no actualizar deps el día que salen
    - Dependabot activado en cada repo

  ▸ Datos:
    - Ley 1581/2012 (protección datos Colombia)
    - Política de privacidad linkeada en todas las apps
    - No almacenar datos sensibles en localStorage
    - Sanitizar inputs en API (validation rules Laravel)

  3.2 CALIDAD DE CÓDIGO
  ─────────────────────

  ▸ TypeScript:
    - strict: true en todos los proyectos
    - No @ts-ignore sin comentario justificando
    - Interfaces en archivo lib/types.ts centralizado

  ▸ Linting:
    - ESLint configurado en cada proyecto
    - npm run lint = 0 errores antes de merge
    - No warnings suprimidos sin justificación

  ▸ Build:
    - npm run build debe compilar sin errores
    - No dead code (tree-shaking verificado)

  ▸ Tests:
    - Backend: PHPUnit feature tests por módulo
    - Frontend web: Playwright E2E (mínimo smoke + regression)
    - Mobile: Jest unit tests + Detox E2E (fase avanzada)
    - Mínimo 80% de flujos críticos cubiertos

  ▸ Git:
    - Conventional commits (feat:, fix:, test:, sec:, docs:)
    - Branch protection en main
    - No force push
    - Tags semánticos para releases (v1.0.0, v1.1.0)

  3.3 DISEÑO Y UX
  ────────────────

  ▸ Design System compartido:
    - Primary: #d1007f, Delivered: #12a85f, Route: #1f86ff
    - Pending: #ff8616, Issue: #e72256
    - Dark bg: #0f0f23, Surface: #1a1a2e, Border: #2a2a3e
    - Text: #e0e0e0, Muted: #94a3b8

  ▸ UX obligatorio:
    - Skeleton loading (no spinners solos)
    - Empty states con ícono + texto + CTA
    - Error states con retry
    - Animaciones: fade-in, scale-95 en botones
    - Dark mode completo (no parcial)
    - Responsive mínimo 375px (mobile-first)
    - Touch targets mínimo 44px (WCAG)

  6.2 VISIÓN DEL PRODUCTO
  ───────────────────────

  El portal del cliente permite a los CLIENTES de Danhei Express
  (empresas/personas que envían paquetes) consultar sus envíos,
  ver facturas, rastrear paquetes y contactar soporte.

  NO es un panel administrativo. Es una interfaz de consulta y
  autoservicio.

  6.3 PANTALLAS — ESPECIFICACIÓN
  ──────────────────────────────

  LOGIN
    - Logo Danhei + fondo gradiente dark
    - Email + contraseña
    - "Olvidé contraseña" (link a WhatsApp soporte)
    - Botón "Ingresar"
    - Token Sanctum → localStorage (migrar a HttpOnly después)

  DASHBOARD
    - Saludo: "Buen día, {nombre empresa}"
    - Cards resumen:
      * Envíos activos (en ruta)
      * Envíos entregados (mes actual)
      * Balance pendiente (deuda)
    - Últimos 5 envíos con status badge
    - Botón "Nuevo envío" → navega a /envios?nuevo=true

  ENVÍOS
    - Tabla/lista de todos los envíos del cliente
    - Filtros: estado, fecha
    - Buscador por código guía
    - Click → modal detalle con timeline
    - Botón "Nuevo envío" → modal formulario
    - Tracking code para compartir

  FINANZAS
    - Balance actual (total por cobrar al cliente)
    - Historial de facturas/cobros
    - Status: pagado / pendiente / vencido
    - Detalle de cada envío con costos

  NOTIFICACIONES
    - Lista de notificaciones del sistema
    - Marcar como leída
    - Tipos: envío entregado, novedad, factura

  PERFIL
    - Datos de la empresa/persona
    - Direcciones guardadas (CRUD)
    - Cambiar contraseña
    - Cerrar sesión

  SOPORTE
    - FAQ básico (3-5 preguntas frecuentes)
    - Formulario PQR
    - Link WhatsApp directo
    - Historial de PQR (si se implementa backend)

  6.4 CONTRATO API — ENDPOINTS QUE USA P14
  ─────────────────────────────────────────

  Auth:
    POST /api/login                  → Login
    POST /api/logout                 → Cerrar sesión
    GET  /api/me                     → Datos del usuario
    PUT  /api/me                     → Editar perfil
    PUT  /api/me/password            → Cambiar contraseña

  Envíos:
    GET  /api/shipments              → Lista envíos (filtro por client_id automático)
    GET  /api/shipments/{id}         → Detalle envío
    POST /api/shipments              → Crear envío
    GET  /api/track?code=XXXX        → Tracking público

  Finanzas:
    GET  /api/clients-receivable     → Balance del cliente
    GET  /api/clients/{id}           → Detalle financiero

  Direcciones:
    POST   /api/clients/{id}/addresses     → Crear dirección
    PUT    /api/client-addresses/{id}      → Editar
    DELETE /api/client-addresses/{id}      → Eliminar

  Notificaciones:
    GET  /api/notifications          → Lista
    POST /api/notifications/read-all → Marcar leídas

  ENDPOINT NUEVO NECESARIO:
    GET /api/client/my-dashboard     → Dashboard del cliente autenticado
    (combina: envíos activos + entregados mes + balance)

  6.5 PERMISOS — ROL "client"
  ────────────────────────────

  El cliente necesita un rol "client" con permisos mínimos:

    shipments.view      (solo los suyos, filtrado automático)
    shipments.create    (crear envíos)
    clients.view        (solo su perfil)
    clients.edit        (editar su perfil)

  NO debe tener: drivers.*, users.*, financial.settle, routes.*

  NOTA: El backend debe filtrar automáticamente por client_id
  asociado al usuario autenticado. El cliente NUNCA debe ver
  envíos de otro cliente.
