1. INTRODUCCIÓN Y CONTEXTO
===========================
Durante el desarrollo local de la plataforma P16-DHE-Admin-Web, se ha utilizado SQLite como motor de base de datos debido a su rapidez y portabilidad. Sin embargo, para el entorno de producción (cPanel), el motor objetivo es MySQL (base de datos 'danheiex_production').

La transición de SQLite a MySQL suele introducir riesgos debido a discrepancias en la sintaxis SQL cruda, funciones de fecha, manejo de tipos booleanos, restricciones de integridad referencial y directivas de agrupamiento estricto. Este reporte técnico evalúa en detalle estos puntos para permitir una toma de decisiones informada antes del despliegue.


2. DIFERENCIAS CLAVE ENTRE SQLITE Y MYSQL Y SU ESTADO EN EL PROYECTO
======================================================================

A. FUNCIONES DE FECHA Y HORA (DATE/TIME FUNCTIONS)
--------------------------------------------------
* El Problema: SQLite utiliza 'strftime' y 'date()' para formatear fechas y extraer horas. MySQL utiliza 'DATE_FORMAT()' e 'HOUR()'. Usar la sintaxis de un motor en el otro causará un fallo fatal (error 500) en producción.
* Evaluación en P16: El único punto que requería procesamiento de fechas crudo en base de datos era la consulta de estadísticas por hora en el Dashboard.
* Estado: RESUELTO. En 'ShipmentController.php' (líneas 290-299), ya se implementó lógica adaptativa que detecta el driver activo (DB::getDriverName()) y aplica la sintaxis correcta dinámicamente:
  - MySQL/MariaDB: CAST(DATE_FORMAT(created_at, '%H') AS UNSIGNED)
  - PostgreSQL: EXTRACT(HOUR FROM created_at)::int
  - SQLite (por defecto): CAST(strftime('%H', created_at) AS INTEGER)

B. AGRUPAMIENTO ESTRICTO (ONLY_FULL_GROUP_BY)
---------------------------------------------
* El Problema: MySQL por defecto viene configurado con 'ONLY_FULL_GROUP_BY' activado. Esto impide seleccionar columnas no agregadas que no estén presentes en la cláusula 'GROUP BY'. SQLite es sumamente permisivo con esto, lo que hace que consultas que funcionan en desarrollo fallen inmediatamente en producción MySQL (Error #1055).
* Evaluación en P16: Se auditaron todas las consultas que implementan 'groupBy()' (ReportController y ShipmentController).
* Estado: COMPATIBLE. Todas las agrupaciones se realizan sobre la misma columna que se selecciona (ej. agrupar por 'status' y contar registros, o agrupar por 'hour'). Cumplen al 100% con el estándar ANSI SQL requerido por MySQL estricto.

C. COLUMNAS Y CONSULTAS JSON
----------------------------
* El Problema: SQLite almacena JSON como texto plano y, aunque soporta operaciones JSON básicas, la sintaxis difiere sustancialmente de la sintaxis nativa de MySQL (JSON_EXTRACT o el operador '->' de Laravel).
* Evaluación en P16: Se detectaron columnas de tipo JSON en las migraciones de envíos ('metadata'), auditoría ('old_values', 'new_values') y zonas ('metadata').
* Estado: COMPATIBLE. El código de la aplicación no realiza búsquedas indexadas directas dentro del JSON en la base de datos (no hay consultas del tipo '->where("metadata->gps", ...)'). El JSON solo se lee y escribe como un bloque completo (cast de Laravel a array/objeto). Esto es perfectamente compatible con ambos motores.

D. BÚSQUEDAS DE TEXTO (LIKE) Y COLLATION
-----------------------------------------
* El Problema: En SQLite, la sentencia 'LIKE' es insensible a mayúsculas/minúsculas de manera predeterminada para caracteres ASCII. En MySQL, la sensibilidad depende de la colación (collation) de la tabla.
* Evaluación en P16: Las búsquedas en la aplicación (ej. buscar clientes por nombre o envíos por código) se realizan con 'like' a través de Eloquent.
* Estado: COMPATIBLE. La base de datos de producción MySQL está configurada para usar la colación 'utf8mb4_unicode_ci', la cual es insensible a mayúsculas y minúsculas ('_ci' significa Case Insensitive). Por ende, el comportamiento en producción será idéntico al de desarrollo.

E. RESTRICCIONES DE CLAVES FORÁNEAS (FOREIGN KEYS)
--------------------------------------------------
* El Problema: SQLite no valida de forma estricta las claves foráneas por defecto en entornos de desarrollo a menos que se fuerce la directiva. MySQL es sumamente riguroso: si intentas insertar un registro cuyo ID relacionado no existe, la base de datos abortará la transacción.
* Evaluación en P16: Se ejecutó el set completo de pruebas automatizadas ('php artisan test').
* Estado: COMPATIBLE. Las 173 pruebas automatizadas (que realizan más de 612 aserciones y ejecutan seeders en bases de datos en memoria con claves foráneas activadas) pasaron con éxito (0 fallos). Esto garantiza que la estructura relacional y los seeders de la aplicación respetan perfectamente la integridad de datos.


3. ¿POR QUÉ Y EN QUÉ VAMOS A NECESITAR ESTO EN EL DEPLOY?
=========================================================
Realizar esta auditoría nos ayuda a:
1. Evitar "Sorpresas de Producción" (Error 500): El escenario clásico donde el desarrollador dice "en mi máquina funciona perfectamente" y, al subirlo a cPanel, la pantalla del dashboard o las búsquedas se rompen debido a incompatibilidades sintácticas.
2. Validar que la migración inicial de tablas ('php artisan migrate') se ejecute sin errores de tipos de columna o restricciones mal formadas.
3. Asegurar que los seeders de producción no fallen por violaciones de claves foráneas.


4. CONCLUSIÓN Y RECOMENDACIÓN PARA LA TOMA DE DECISIÓN
======================================================
* RIESGO DE INCOMPATIBILIDAD ACTUAL: MUY BAJO (cercano a 0%).
* RAZÓN: El código del backend está extraordinariamente limpio. Se utiliza Eloquent ORM de Laravel para el 99.5% de las interacciones, el cual traduce automáticamente las consultas al dialecto de la base de datos activa. La única consulta compleja (gráficos del dashboard) ya fue blindada previamente con detección del driver de base de datos.

RECOMENDACIÓN FINAL:
No es necesario invertir tiempo en modificar código o reescribir queries para MySQL en el backend, dado que ya se encuentra optimizado y blindado.
La única acción necesaria antes del deploy real es crear una base de datos MySQL vacía (ya sea localmente en su entorno de desarrollo o directamente en un entorno de pruebas en cPanel) y ejecutar:
  'php artisan migrate:fresh --seed'
Esto confirmará en un 100% que la base de datos MySQL inicializa, crea sus tablas, y puebla los datos iniciales de forma correcta sin restricciones de claves foráneas rotas.
