# Mi conclusión: **sí, esta idea tiene mucho sentido para Danhei**

De hecho, considero que puede convertirse en uno de los canales principales de entrada de pedidos de Danhei Express.

Pero hay una diferencia fundamental:

> **No debemos hacer un simple chatbot que pregunte “nombre”, luego “dirección”, luego “teléfono”, dato por dato.**

Eso funciona técnicamente, pero operativamente se vuelve lento, genera conversaciones incompletas y aumenta la probabilidad de datos mal escritos.

La arquitectura profesional sería:

# **Danhei Recogidas por WhatsApp**

### WhatsApp Business Platform + WhatsApp Flows + API Danhei + Panel Administrativo

WhatsApp Flows permite presentar formularios estructurados dentro de la propia conversación, sin obligar al cliente a salir hacia una página externa. Meta contempla Flows precisamente para procesos como reservas, captura de información y operaciones conectadas con sistemas empresariales. Además, un Flow puede comunicarse con un endpoint del backend para consultar y validar datos en tiempo real. ([Desarrolladores de Facebook][1])

# 1. La experiencia que yo diseñaría

El cliente entra al WhatsApp actual de Danhei y escribe algo como:

> Hola
> Necesito una recogida

El sistema reconoce la intención y responde:

**Hola, Diego 👋**
¿Qué deseas hacer?

`📦 Solicitar recogida`
`🔍 Rastrear paquete`
`💬 Hablar con un asesor`

Cuando toca:

## `📦 Solicitar recogida`

Se abre dentro de WhatsApp el formulario Danhei.

---

## Pantalla 1 — ¿Dónde recogemos?

Para un cliente nuevo:

**Dirección de recogida**

* Dirección
* Complemento: local, torre, apartamento, bodega
* Barrio
* Nombre de quien entrega
* Teléfono de contacto
* Instrucciones al piloto

Pero para un cliente frecuente sería mucho mejor:

**¿Dónde recogemos?**

`📍 Mi dirección habitual`
`📍 Calle 13 #...`
`➕ Agregar nueva dirección`

Esto es muy importante.

Un cliente frecuente de Danhei **no debería escribir la misma dirección todos los días**.

---

# 2. La dirección no debe ser simple texto

Este es uno de los puntos donde debemos ser más exigentes técnicamente.

Actualmente el cliente puede escribir:

> kr 89a # 12 34 al lado de la panadería

Eso puede terminar en el panel administrativo como una dirección difícil de enrutar.

La propuesta profesional sería:

1. El cliente escribe la dirección.
2. Danhei la geocodifica.
3. Guardamos:

   * dirección escrita;
   * dirección normalizada;
   * latitud;
   * longitud;
   * localidad/municipio;
   * zona operativa;
   * nivel de confianza de la geocodificación.
4. El sistema determina automáticamente si está dentro de cobertura.

Google dispone de APIs de autocompletado y geocodificación que convierten direcciones en coordenadas y permiten limitar búsquedas geográficamente. ([Google for Developers][2])

En Danhei esto permitiría validar automáticamente:

* Bogotá urbana;
* Soacha;
* Madrid;
* Mosquera;
* Funza;
* Cota;
* Cajicá;
* Chía;
* Zipaquirá.

Y evitaría que el administrador tenga que descubrir manualmente que una recogida está fuera de cobertura.

---

# 3. Pantalla 2 — Información de los paquetes

Aquí no debemos cometer el error de pensar que:

> 1 recogida = 1 paquete.

Para Danhei la estructura correcta sería:

## Una solicitud de recogida

puede contener:

## Uno o varios paquetes

Por eso el formulario preguntaría:

**¿Cuántos paquetes debemos recoger?**

`1` `2` `3` `4` `5+`

Luego:

### Paquete 1

* Tipo de paquete
* Tamaño aproximado
* Peso aproximado
* ¿Es frágil?
* ¿Tiene alguna condición especial?

Opciones rápidas:

`📄 Sobre/documento`
`📦 Paquete pequeño`
`📦 Paquete mediano`
`📦 Paquete grande`

No le pediría al cliente medidas exactas desde el primer día, porque aumentaría el abandono.

La primera versión debe capturar **solo los datos necesarios para operar correctamente**.

---

# 4. Pantalla 3 — ¿A dónde va?

Aquí hay dos escenarios.

## Escenario A: recogida simple

El cliente solo necesita que Danhei recoja la mercancía.

Entonces:

* cantidad de paquetes;
* dirección de recogida;
* franja de recogida.

## Escenario B: el cliente ya tiene todos los datos del envío

Entonces puede diligenciar:

* destinatario;
* teléfono;
* dirección;
* barrio;
* complemento;
* contenido;
* valor declarado;
* tipo de servicio;
* valor a recaudar si es contraentrega.

En este caso el sistema podría hacer algo mucho más poderoso:

> Crear la recogida **y las guías automáticamente**.

Ahí es donde la integración deja de ser un simple formulario y empieza a ahorrar trabajo administrativo real.

# 5. Pantalla 4 — Tipo de servicio

Por ejemplo:

**Selecciona el servicio**

`⚡ Servicio hoy`
`📦 Paquetería`
`💰 Contraentrega`

Para contraentrega:

**Valor que debemos recaudar**

`$ __________________`

Aquí el sistema debería aplicar reglas de negocio:

* valor mínimo;
* valor máximo;
* cliente autorizado;
* forma de recaudo;
* datos de liquidación.

Esto además puede conectarse posteriormente con el sistema de recaudo directo que hemos venido analizando para evitar que el dinero pase por la cuenta personal del piloto.

---

# 6. Pantalla 5 — Jornada de recogida

Danhei ya tiene una lógica operacional de jornadas.

El formulario no debería pedir libremente:

> ¿A qué hora quiere que recojamos?

Eso destruye la planificación logística.

El sistema debería mostrar únicamente las opciones operativamente posibles.

Por ejemplo:

### Si todavía hay capacidad:

`🌤 Primera jornada`
`🌆 Segunda jornada`

### Si la primera ya cerró:

`🌆 Próxima recogida disponible`

### Si ambas cerraron:

`📅 Programar para mañana`

Y algo todavía más avanzado:

> Si una zona ya alcanzó la capacidad máxima de recogidas, el backend simplemente deja de ofrecer esa franja.

WhatsApp Flows puede conectarse con un backend para ofrecer información dinámica; Meta ejemplifica esta arquitectura con procesos que consultan disponibilidad en tiempo real. ([WhatsApp for Business][3])

# 12. Una regla crítica: evitar pedidos duplicados

Imagine esto:

1. El cliente confirma.
2. La API crea la recogida.
3. La respuesta tarda.
4. WhatsApp repite el evento.
5. La API vuelve a crear la misma recogida.

Resultado:

> Dos pilotos podrían salir por los mismos paquetes.

Para evitarlo implementaría:

* `meta_message_id` único;
* `flow_submission_id` único;
* claves de idempotencia;
* transacciones de base de datos.

Una misma confirmación **jamás puede crear dos recogidas**.

Esta decisión es fundamental para que Danhei pueda crecer.

# 15. No empezaría con inteligencia artificial

Mi recomendación técnica es muy clara:

## Primera versión

**Flujo determinista y controlado.**

* botones;
* formularios;
* validaciones;
* reglas;
* estados.

## Segunda versión

IA solamente para comprender frases como:

> Necesito que recojan cinco cajas en mi local mañana.

La IA interpreta la intención y propone:

> Entendí que deseas solicitar una recogida de 5 paquetes. ¿Es correcto?

Pero **la IA nunca debe crear por sí sola una operación logística definitiva**.

La confirmación estructurada debe seguir siendo obligatoria.

Esto elimina alucinaciones y pedidos mal creados.

# 16. Seguridad de producción

Yo exigiría desde la primera versión:

### En WhatsApp

* validación de webhooks;
* validación de firmas;
* secretos fuera del código;
* tokens rotables;
* registro de eventos.

### En la API

* idempotencia;
* rate limiting;
* validación server-side;
* auditoría;
* control antifraude;
* detección de duplicados.

### En los Flows

* endpoint cifrado;
* tokens con expiración;
* validación del contexto;
* observabilidad.

Meta proporciona mecanismos específicos para configurar cifrado del endpoint de Flows y consultar métricas como disponibilidad y errores del endpoint. ([postman.com][5])

# 17. Protección de datos en Colombia

Aquí se recopilarán:

* nombres;
* teléfonos;
* direcciones;
* ubicaciones;
* destinatarios;
* historial de envíos.

Por tanto, no deberíamos lanzar el sistema sin adaptar el aviso y la política de tratamiento de datos de Danhei.

La Ley 1581 de 2012 exige que el tratamiento tenga finalidad informada y establece obligaciones de seguridad para evitar adulteración, pérdida o acceso no autorizado. ([Función Pública][6])

En la última pantalla del Flow incluiría:

☑️ **Autorizo el tratamiento de mis datos para gestionar la recogida y prestación del servicio.**

Con acceso a:

`Ver política de tratamiento de datos`

# 18. ¿Qué pasa con el WhatsApp actual de Danhei?

Este punto hay que manejarlo con mucho cuidado.

Meta añadió soporte para incorporar números existentes de WhatsApp Business a Cloud API y existe una modalidad de **Coexistence** que puede permitir mantener el uso de la aplicación y obtener acceso a Cloud API en el mismo número. Sin embargo, la ruta exacta de incorporación y la elegibilidad deben verificarse antes de intervenir el número de producción, porque algunos procesos de incorporación pueden alterar los dispositivos vinculados. ([Desarrolladores de Facebook][7])

Mi criterio sería:

> **No tocar el +57 311 220 6587 hasta probar toda la integración con un número de prueba.**

Orden correcto:

```text
1. Número de prueba Meta
2. Flow de recogida
3. Webhook
4. API Danhei
5. Panel administrativo
6. Pruebas integrales
7. Auditoría del número actual
8. Migración o coexistencia
```

# 19. Costos de mensajería

Actualmente Meta cobra la plataforma por mensaje entregado según categoría y mercado. Las interacciones iniciadas por el usuario abren una ventana de servicio de 24 horas; el esquema actual permite respuestas de servicio gratuitas en esa ventana. Meta ya ha anunciado cambios de precios para mensajes de utilidad a partir del **1 de octubre de 2026**, por lo que el módulo debe aislar la lógica de mensajes y plantillas para poder adaptarse sin modificar el núcleo de pedidos. ([Desarrolladores de Facebook][8])

Esto refuerza otra decisión:

> No debemos llenar WhatsApp de notificaciones innecesarias.

Yo enviaría solo eventos de valor:

1. Recogida solicitada.
2. Recogida confirmada.
3. Piloto en camino.
4. Paquetes recogidos.
5. Incidencia real.
